# 기능 확인 및 테스트 가이드

## 기대하신 기능 vs 실제 구현

| 기대 | 실제 |
|------|------|
| AI가 고독사 위험이 **높아지면 자동**으로 보호자에게 카메라 캡처 | **아님.** AI 확률이 높아도 사진은 자동 전송되지 않음 |
| 캘린더에 **AI 위험도 예측** 저장 | **아님.** 캘린더는 보호자가 직접 입력하는 **케어 일정 메모**(앱 메모리, 서버 미저장) |
| 위험 시 보호자가 사진 확인 | **부분 구현.** 아래 「긴급 사진」 플로우 |

### 실제로 있는 기능

1. **AI 위험 분석**  
   - 피보호자가 좌표를 보낼 때마다 Groq 분석 → `risk_assessment` 테이블 저장  
   - 보호자 **홈** 탭: WebSocket + API로 **최신** 위험도·요약 문구 표시 (캘린더 X)

2. **위험 알림(Crisis)** — 규칙 기반, AI 확률과 별개  
   - `TOILET_OVERFLOW`: 화장실 체류 시간 > 평소×3  
   - `LETHARGY`: 최근 10회 좌표 활동 반경이 평소의 50% 미만  
   - 보호자 **알림** 탭에 활성 위험 목록 표시

3. **긴급 실시간 사진** (Crisis가 **활성**일 때만)  
   - 보호자가 **「긴급 실시간 사진 요청」** 버튼을 눌러야 함 (자동 X)  
   - 피보호자 앱이 촬영 → 서버 메모리 1회 업로드 → 보호자가 1회 열람 후 삭제

4. **캘린더**  
   - 보호자가 날짜별로 「케어 스케줄」 텍스트를 수동 추가 (AI 연동 없음)

## 테스트 환경 구성

### 1. 백엔드 (필수)

```bash
docker compose up -d
cp application-secrets.properties.example application-secrets.properties   # AI_API_KEY 설정
./gradlew bootRun
```

- DB: `localhost:3306/godoksa_db`  
- AI: `application-secrets.properties` 또는 환경변수 `AI_API_KEY`  
- IntelliJ Run에 **빈 `AI_API_KEY`** 가 있으면 제거

### 2. 피보호자 앱

```bash
cd app
flutter run -d macos   # 또는 Android 에뮬레이터
```

1. **피보호자** 선택 → 연결 코드 발급 → **모니터링 시작**  
2. 모니터링 화면에서 카메라 허용  
3. 좌표가 5초마다 서버로 전송되는지 확인 (`좌표 출처`, `분석` 문구)

### 3. 보호자 앱 (두 번째 실행)

**같은 PC**에서 터미널을 하나 더 열거나, 다른 기기에서 **같은 백엔드 주소**로 접속:

```bash
cd app
flutter run -d macos
# 실기기/다른 PC: --dart-define=API_BASE_URL=http://<서버IP>:8080
```

1. **보호자** 선택 → 피보호자 **6자리 코드** 입력 → 연결  
2. **홈**: 고독사 위험도·AI 요약 (피보호자가 좌표 보낼 때 갱신)  
3. **알림**: Crisis 발생 시 목록 표시  
4. **캘린더**: 수동 일정만 (AI와 무관)

> 피보호자·보호자 앱이 **서로 다른 PC의 localhost**만 바라보면 연동 안 됨. 팀 테스트 시 **한 명 PC IP:8080** 또는 공유 ngrok DB + 각자 백엔드 중 **동일 서버 URL** 사용.

## 시나리오 A: AI 위험도만 보기 (가장 쉬움)

1. 백엔드 + 피보호자 모니터링 실행  
2. 보호자 앱 연결  
3. 피보호자 화면에서 **거실** 유지, 카메라/좌표 전송  
4. 보호자 **홈**에서 `고독사 위험도`, `분석` 문구 변경 확인  

DB 확인(선택):

```sql
SELECT probability, ai_summary, created_at
FROM risk_assessment
ORDER BY created_at DESC
LIMIT 5;
```

---

## 시나리오 B-0: AI 위험도 → 보호자 자동 알림 (실제 경로)

좌표 전송 → AI 분석 → 확률 ≥ `monitoring.ai.crisis-threshold`(기본 **60%**) → Crisis `AI_HIGH_RISK` + WebSocket.

### 테스트 절차

1. `application.properties`에서 임시로 임계값 낮추기 (쉽게 트리거):

```properties
monitoring.ai.crisis-threshold=0.2
```

2. 백엔드 **재시작** + 피보호자·보호자 앱 연결  
3. 피보호자 **모니터링** 유지 (5초마다 좌표+AI)  
4. 화면 **고독사 확률**이 20% 이상이 되는 순간 → 피보호자 스낵바 「보호자 자동 알림 조건 충족」  
5. 보호자: 빨간 스낵바 + **알림** 탭 → **「AI 고독사 위험 경고」**  

같은 `AI_HIGH_RISK`가 이미 활성이면 중복 알림은 없음 → 다시 테스트하려면 보호자 **알림** 탭에서 조치 완료(해제) 후 재시도.

테스트 후 임계값을 `0.6`으로 되돌리세요.

### 수동 버튼 (경로만 확인)

피보호자 하단 **「알림 경로만 수동 테스트」** — AI와 무관, UI 연결 확인용.

## 시나리오 B-0b: (구) 수동 테스트 버튼

위 수동 버튼과 동일. **실제 AI 자동 알림 검증에는 B-0 사용.**

## 시나리오 B: 긴급 사진 (Crisis → 수동 요청)

### B-1. 화장실 Crisis 빠르게 만들기

피보호자 앱:

1. 모니터링 화면에서 **화장실** 세그먼트 선택  
2. 체류 시간이 쌓이기 전에, 아래 스크립트로 한 번에 Crisis 생성 가능:

```bash
./scripts/test-crisis-flow.sh toilet <LOGIN_CODE> <HARDWARE_ID>
```

(`LOGIN_CODE` / `HARDWARE_ID`는 피보호자 앱·연결 코드 화면 기준)

또는 앱에서 **화장실** 선택 후 **20분 이상** 대기 (평소 20분×3 = 60분 필요 — 테스트엔 스크립트 권장)

### B-2. 무기력(LETHARGY) Crisis

동일 좌표를 10회 이상 전송:

```bash
./scripts/test-crisis-flow.sh lethargy <LOGIN_CODE> <HARDWARE_ID>
```

### B-3. 사진 요청 ~ 열람

1. 보호자 **알림** 탭 → 활성 위험 있는지 확인  
2. 보호자 **홈** → 「긴급 위험 활성 — 사진 열람 가능」  
3. **긴급 실시간 사진 요청** 버튼 탭  
4. 피보호자 앱: 스낵바 「보호자 사진 요청」→ 자동 촬영·업로드  
5. 보호자: 사진 다이얼로그 표시 (1회)

## 시나리오 C: 캘린더

보호자 앱 → **캘린더** 탭 → 날짜 선택 → 일정 입력 → 저장  
→ 앱을 재시작하면 **메모가 사라질 수 있음** (서버/DB 미연동)

---

## 자주 막히는 지점

| 증상 | 원인 |
|------|------|
| 사진 버튼 비활성 | 활성 `Crisis` 없음 → B 시나리오로 Crisis 먼저 생성 |
| 보호자 위험도 안 바뀜 | 다른 localhost 백엔드 / WebSocket 미연결 |
| AI 문구가 항상 「거실에서 규칙적인…」 | Groq 429 → 모델·키·백엔드 재시작 |
| 피보호자 사진 안 올라감 | 카메라 미초기화, 모니터링 화면이 아님, 권한 거부 |

## API로 직접 확인 (curl)

```bash
# 활성 위험
curl -s "http://127.0.0.1:8080/api/crisis/active?loginCode=123456"

# 최신 AI 평가
curl -s "http://127.0.0.1:8080/api/risk/latest/123456"

# 사진 요청 (Crisis 있을 때만 200)
curl -s -X POST "http://127.0.0.1:8080/api/guardian/photo-request/123456"
```
